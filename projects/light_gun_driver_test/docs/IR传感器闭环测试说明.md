# IR传感器闭环测试说明

本文档对应工程：

- `projects/light_gun_driver_test`

对应源码文件：

- `sdk_overlay/custom/ir_test.c`
- `sdk_overlay/custom/ir_test.h`

## 1. 这套测试在做什么

这套 IR 测试是参考：

- `ref_projs/OpenFIRE-Firmware-ESP32/docs/IR_SENSOR_WIRELESS_ANALYSIS.md`
- `ref_projs/OpenFIRE-Firmware-ESP32/lightgun/lib/DFRobotIRPositionEx/`

做的一版“方便你二次开发”的最小闭环。

目标不是一次性把 OpenFIRE 全量高级几何算法全部移植，而是先把下面这条链路跑通：

1. TR531x I2C 初始化
2. Wii IR Camera 兼容模块寄存器初始化
3. Basic frame 双帧稳定读取
4. 解包出 4 个原始红外点
5. 做一版简单但稳定的坐标解算
6. 按固定周期输出日志，便于你对照串口和示波器继续扩展

这样你后续再做：

- 四点排序增强
- 透视修正
- 校准参数持久化
- 与 USB 鼠标联动

就有一个已经能工作的底座。

## 2. 当前实现范围

当前实现了：

1. I2C 主机初始化
2. IR 传感器寄存器初始化
3. `0x36` basic frame 读取
4. 双帧比对式 `atomic read`
5. 4 点解包和 `seen mask`
6. “左右点中心 + 线性映射”的简化坐标解算
7. 简单一阶平滑
8. 软件定时器轮询闭环测试
9. 回放模式，不接真传感器也能先验证解算链路
10. 精简日志模式，避免串口被每帧日志刷满
11. 单点降级解算，只有 1 个点时也能先给出近似坐标
12. 当 IR 已接入真实 USB 鼠标链路时，正常坐标日志自动降到约 5 秒一条
13. 在 basic frame 解包阶段过滤明显无效点，例如常见的 `(1023,1023)` 空值哨兵
14. 当出现原子读回退或整帧无有效点时，额外打印 11 字节原始 basic frame 十六进制日志

当前没有实现：

1. OpenFIRE 的高级四点重建
2. 高级透视修正
3. 一欧元滤波
4. 外部 24/25MHz Wii Camera 时钟输出

最后这一点要特别注意：

- 如果你手上的模块是“裸 Wii Camera”，通常还需要外部 24MHz/25MHz 时钟
- 现在这份测试代码只负责 I2C 初始化和读点，不负责给相机额外送高速时钟
- 如果没有这个时钟，常见现象就是：I2C 初始化成功，但始终读不到有效点

## 3. 初始化流程

当前初始化流程基本对齐 `DFRobotIRPositionEx::begin()`：

1. 配置 I2C pinmux
2. 初始化 I2C 主机
3. 写 `0x30 = 0x01`
4. 写灵敏度相关寄存器 `0x06 / 0x08 / 0x1A`
5. 写 `0x33 = 0x11`，切到 basic data format
6. 写 `0x30 = 0x08`，启动相机

程序员视角可以把它理解成：

- 前几步是在“把相机唤醒并配置工作模式”
- 后几步是在“告诉相机以后按 basic frame 格式吐数据”

## 4. 为什么要做双帧稳定读取

OpenFIRE 里会调用 `basicAtomic()`，不是单纯读一帧就用。

原因很简单：

- 相机内部也在不断刷新
- 如果主控刚好在相机刷新中途去读，有概率读到半新半旧的一帧
- 这样坐标会抖得很厉害

当前测试里的处理方式是：

1. 连续读两帧 basic frame
2. 比较除头字节外的数据是否一致
3. 一致就认为这次数据稳定
4. 如果连续多次都不一致，最后退化使用最新一帧并打印状态

这一步验证的是：

- I2C 读取是否稳定
- 数据是否容易撕裂
- 你的传感器时钟和供电是否足够稳

## 5. 当前坐标解算是怎么做的

当前实现已经从最初的“左右点中心”升级成一版更接近 OpenFIRE square 流程、但仍然便于调试的最小可用算法：

1. 先读取并解出最多 4 个原始 IR 点
2. 如果看到 4 点，直接给 4 点排序并识别 `TL / TR / BL / BR`
3. 如果看到 3 点，用平行四边形关系补第 4 点，再排序识别四角
4. 如果只看到 2 点或 1 点，优先使用上一帧已经锁定的四角身份做退化跟踪
5. 用四角中心作为 `center_x / center_y`
6. 再把中心点从相机坐标线性映射到屏幕坐标
7. 最后做一次简单平滑

这样做的优点是：

1. 已经能开始验证“上下左右四角身份”而不只是左右边界
2. 仍然比完整 OpenFIRE 高级几何更容易调日志
3. 后面要继续接透视、滤波或更强缺点补偿时，接口不用重做

它仍然不是 OpenFIRE 全量高级几何，但已经不再是纯粹的“最左最右 + y 平均”测试版。

## 6. 测试模式说明

Kconfig 里提供了 4 种模式。

### 6.1 Init only

验证点：

- I2C 能不能初始化成功
- 寄存器写入有没有报错

适合：

- 第一次上板
- 先看连线是否通

### 6.2 Raw point stream

验证点：

- `seen mask` 是否正常变化
- 4 个点的原始坐标是否能读出来
- 双帧读取是否经常 mismatch

适合：

- 调相机供电
- 调灵敏度
- 看有没有点但坐标还没解好

### 6.3 Live solve

验证点：

- 原始点是否能稳定变成屏幕坐标
- 平滑参数是否合适
- 左右/上下方向是否正确

适合：

- 做最基础的准星联调
- 后续对接 USB 鼠标前先看日志

### 6.4 Replay solve

验证点：

- 不接真实传感器时，解算函数本身是否能按预期工作
- 坐标映射和日志格式是否正确

适合：

- 先开发算法框架
- 板子没接 IR 时先通逻辑链

## 7. 调试增强开关说明

## 7.5 `(1023,1023)` 日志代表什么

如果你在旧日志里看到：

- `seen=4`
- 但 `raw p0..p3` 大量是 `(1023,1023,1)`

这通常不表示“真的看到了 4 个点”，而更像是：

1. 相机当前没有锁到有效红外点
2. 或 basic frame 里的空槽位被上层代码误当成有效点

现在这版代码已经在解包阶段把这类哨兵值过滤掉，所以后续更合理的现象应该变成：

- `seen=0`
- 或 `seen=1/2`
- 而不是假 `seen=4`

如果后面又出现：

- `basic hex atomic-fallback: ...`
- `basic hex no-valid-points: ...`

就说明问题在更底层，优先看：

- 传感器模组时钟
- 灯条亮度与距离
- I2C 帧内容本身是否长期空值

## 7.6 空帧日志限流与鼠标上屏门槛

为了方便联调，当前又补了两条保护：

1. `basic hex no-valid-points` 连续出现时不再每帧都打印
2. USB live mouse 模式下，只有 `seen >= 2` 才认为“在屏内”

这意味着：

- `seen=1` 仍然会保留调试解算日志
- 但不会再被当成稳定鼠标上屏依据
- 连续 `00 ff ff ff ff ff ff ff ff ff ff` 也不会把串口刷满

## 7.7 四角几何合法性过滤

为了减少“看起来有 3 点/4 点，但实际上几何关系不对”的误判，当前在四角排序后又增加了一层合法性检查。

主要会过滤这些情况：

1. 上下边长度差异过大
2. 左右边高度差异过大
3. 四边形面积过小，接近共线
4. 上下或左右平均位置关系反了

这样做的目的不是追求最强鲁棒性，而是先把明显错误的 `4pt / 3pt-recon / 2pt-track / 1pt-track` 结果挡掉，避免后续鼠标或坐标日志被假矩形拖偏。

另外，当前 `seen=1` 已经进一步收紧为：

- 只保留诊断日志
- 不再作为正式 `solve ok` 有效解算结果

## 7.8 X/Y 方向翻转

当前又补了两个配置项：

- `LIGHT_GUN_IR_INVERT_X`
- `LIGHT_GUN_IR_INVERT_Y`

它们作用在“相机坐标映射到屏幕坐标”这一步，而不是去改原始采点。

结合目前这块板子的现象：

- 正常安装时：左右反、上下对
- 传感器旋转 180 度后：左右对、上下反

这说明当前硬件只需要：

- 打开 `X invert`
- 保持 `Y invert` 关闭

所以当前默认配置已经按这个结论打开了 `LIGHT_GUN_IR_INVERT_X`。

### 7.1 Enable IR compact debug log

作用：

- 不再每一帧都打印完整解算日志
- 改成“状态变化时打印 + 每 N 帧抽样打印”

这样更适合边调灯条、边观察坐标趋势。

### 7.2 IR compact log every N polls

作用：

- 配合 compact log 使用
- 控制抽样日志频率

比如设成 `10`，表示大约每 10 次轮询补打一条摘要日志。

如果已经启用了 `Use live IR solve to drive USB mouse`，正常 `solve ok` 坐标日志会自动进一步降到约 5 秒一条。

### 7.3 Enable single-point degraded solve

作用：

- 当前只看见 1 个 IR 点时
- 不再直接判定“完全无效”
- 而是退化成“单点直接线性映射”

它的意义不是追求精度，而是先让：

- IR 坐标链路
- USB 鼠标链路

能继续联动起来。

### 7.4 IR live mouse 模式下的日志策略

当启用了：

- `Use live IR solve to drive USB mouse`

当前日志策略会自动切换成：

1. 正常 `solve ok` 坐标日志约 5 秒打印一次
2. 出现错误或状态切换时立即打印

这里的“错误或状态切换”包括：

- `basicAtomic read failed`
- `seen=0`
- `valid/degraded` 状态变化

## 8. 顶层多选行为

现在 `light_gun_driver_test` 顶层测试对象已经改成多选：

- `Enable solenoid test`
- `Enable rumble test`
- `Enable IR sensor test`
- `Enable USB simulated closed-loop test`

你勾选哪个，就启动哪个。

如果你把：

- 电磁铁
- 振动马达
- IR
- USB 模拟

都勾上，那么它们会在同一个工程里一起启动，各自跑各自的软件定时器。

## 9. 推荐联调顺序

建议按这个顺序来：

1. 先只开 `IR Init only`
2. 再开 `IR Raw point stream`
3. 确认能稳定读点后切到 `IR Live solve`
4. 再考虑把 IR 解算结果继续接到 USB 鼠标链路

如果第二步一直读不到有效点，优先查：

1. I2C SCL/SDA 接线
2. 传感器供电
3. 地址是否是 `0x58`
4. 模块是否需要外部 24/25MHz 时钟
5. 灵敏度是否合适
