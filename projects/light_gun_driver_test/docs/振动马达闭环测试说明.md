# 振动马达闭环测试说明

本文档对应测试工程：

- `projects/light_gun_driver_test`

对应代码：

- [rumble_test.h](/data/tr531x_proj/projects/light_gun_driver_test/sdk_overlay/custom/rumble_test.h)
- [rumble_test.c](/data/tr531x_proj/projects/light_gun_driver_test/sdk_overlay/custom/rumble_test.c)

## 1. 文档目的

这份文档只聚焦“振动马达闭环测试”。

这里的闭环测试可以这样理解：

- 代码通过 PWM 输出占空比
- 驱动板把 PWM 变成马达的实际转动
- 你通过示波器、日志、听感、手感来观察结果
- 再把实际现象和代码预期一一对照

如果占空比变化、振动时长、振动节奏、强度变化都和代码一致，说明这条振动控制链路是通的。

## 2. 当前测试实现概览

当前实现采用 PWM 控制振动马达，默认参数按当前测试硬件约定：

- PWM 通道：`1`
- PWM 组：`1`
- PWM 引脚：`GPIO1`
- PWM 引脚模式：`42`

这些默认值和测试模式现在都可以通过 `Kconfig` 配置，不需要再手改代码宏。

这里特别说明一下 `GPIO1 PWM1`：

- 当前 SDK 头文件里，`41/42/43` 这一段注释有“正向输出 / COMP 互补输出”混写的情况
- 结合实际排障，`GPIO1 + 通道1` 更适合先用 `42` 来验证正向 PWM 波形
- 如果你当前完全看不到波形，优先先确认这里，而不是先怀疑软件定时器

另外，这个测试工程现在会在启动 rumble 测试前，主动扫描并清理所有残留的 PWM 复用脚：

- 目的是避免你上一轮把 `GPIO5` 设成 PWM，下一轮又把 `GPIO2` 也设成同一路 PWM 后，两个脚同时出波形
- 这是测试工程的保护逻辑，不代表芯片 pinmux 会自动帮你做排他解绑

默认测试策略：

1. 初始化 PWM
2. 默认切回 `GPIO LOW` 空闲态
3. 延时 `1500ms`
4. 如果是诊断模式，直接持续输出固定 PWM
5. 如果是脉冲模式，再启动软件定时器状态机
6. 每轮结束都回到 `GPIO LOW`

默认参数：

- 默认强度：`70%`
- 默认单次振动时长：`120ms`
- 默认两次振动间隔：`180ms`
- 最大占空比钳位：`100%`
- 最大单次持续时间钳位：`300ms`

安全原则：

- 振动马达虽然比电磁铁耐受度高一些，但也不建议长时间满占空比连续驱动
- 所以这里也有持续时间钳位和轮次休息机制

## 3. 测试前接线建议

建议至少观察以下其中之一：

1. 示波器
   最推荐。直接看 PWM 占空比是否变化。

2. 逻辑分析仪
   可以看 PWM 占空比粗略变化，但不如示波器直观。

3. 手感 / 听感
   可以感受到振动强弱和节奏，但不能精确判断占空比。

4. 马达端电压
   可以辅助判断驱动是否起作用，但不如直接看 PWM 输入。

优先观察点：

- PWM 输出脚
- 驱动板 PWM 输入脚
- 振动马达两端电压

## 4. 上电后的预期基础行为

无论哪个测试用例，上电后的基础行为都应该一致：

1. 初始化 PWM
2. 默认空闲态为 `GPIO LOW`
3. 延时 `1500ms`
4. 再开始执行具体测试用例

如果上电后一开始马达就持续震动，优先检查：

- 驱动板是否为低电平有效
- PWM 引脚模式是否配错
- 驱动板默认上电状态是否把马达拉起来了

## 5. 测试用例说明

## 5.1 `RUMBLE_TEST_CASE_CONTINUOUS_DIAG`

作用：

- 这是最推荐先跑的“底层通路诊断模式”

它验证的功能点：

1. `GPIO1` 的 pinmux 是否真的切到了 PWM
2. `PWM channel=1` 和 `group=1` 是否真的启动成功
3. 示波器探头接法是否正确
4. 驱动板 PWM 输入端是否真的能收到持续波形

预期波形：

```text
上电
GPIO LOW 约 1500ms 内完成初始化
随后持续输出固定占空比 PWM
```

预期日志重点：

```text
pwm init done, channel=1 group=1 pin=1 mode=42, idle forced LOW
continuous diagnostic mode: expect stable PWM on GPIO1, mode=42, duty=70%
diagnostic alive: probe GPIO1 for continuous PWM
```

实际观察重点：

- 如果这个模式都没有波形，问题大概率不在状态机
- 优先排查 `pin mode`、探头位置、板级连接、驱动输入脚定义

---

## 5.2 `RUMBLE_TEST_CASE_SINGLE_PULSE_ONCE`

作用：

- 验证最基础的单次振动脉冲

它验证的功能点：

1. 默认是否为空闲低电平
2. 能否从低电平切到目标占空比
3. 振动是否只持续配置的时间
4. 到时后能否回到低电平
5. 测试结束后是否保持静止

预期波形：

```text
上电
GPIO LOW 1500ms
70% 占空比 120ms
GPIO LOW 持续保持
```

预期日志重点：

```text
pwm init done, channel=1 group=1 pin=1 mode=42, idle forced LOW
start delayed by 1500 ms
pulse 1/1 -> duty 70% for 120 ms
pulse 1/1 -> GPIO LOW gap 180 ms
cycle finished, output forced LOW
```

实际观察重点：

- 示波器上应看到一次明显占空比抬高
- 马达只震动一次
- 之后恢复静止

适合场景：

- 第一次确认 PWM 输出和驱动接线
- 第一次确认马达方向和动作是否正常

---

## 5.2 `RUMBLE_TEST_CASE_PERIODIC_PULSE`

作用：

- 验证周期性短振动是否稳定

它验证的功能点：

1. 软件定时器是否稳定
2. 每个周期是否只震动一次
3. 周期结束后是否总能回到 `0%`
4. 长时间运行后是否出现误触发或丢触发

预期波形：

```text
0% 1500ms
70% 120ms
0% 持续到 1000ms 周期结束
70% 120ms
0% ...
循环
```

实际观察重点：

- 周期是否稳定
- 每个周期是否只有一个“振动包”
- 不应出现意外连续振动

适合场景：

- 长时间稳定性测试
- 检查 PWM 输出是否偶发异常

---

## 5.3 `RUMBLE_TEST_CASE_BOUNDED_AUTOFIRE`

作用：

- 验证有限多脉冲连续振动

它验证的功能点：

1. 一轮内多次振动节奏是否正确
2. 脉冲间停顿是否正确
3. 一轮结束后是否强制休息
4. 是否避免马达长时间连续工作

默认配置下，一轮行为如下：

- 连续输出 `5` 次振动
- 每次：
  - `70%` 占空比 `120ms`
  - 然后 `0%` 占空比 `180ms`
- 一轮结束后休息 `1500ms`
- 再开始下一轮

预期波形：

```text
70% 120ms -> 0% 180ms
70% 120ms -> 0% 180ms
70% 120ms -> 0% 180ms
70% 120ms -> 0% 180ms
70% 120ms -> 0% 180ms
0% 保持 1500ms
下一轮
```

实际观察重点：

- 一轮是否正好是 5 次振动
- 轮次之间是否真的停下来
- 振动节奏是否均匀

适合场景：

- 模拟连续振动反馈
- 验证多脉冲状态机

---

## 5.4 `RUMBLE_TEST_CASE_DUTY_SWEEP`

作用：

- 验证占空比和振动强度之间的映射关系

默认实现会按以下顺序输出：

- `25%`
- `50%`
- `75%`
- `100%`

每一级都持续一小段时间，再回到 `0%`，然后进入下一档。

它验证的功能点：

1. 不同占空比是否能正确输出
2. 体感强度是否能随占空比逐级变化
3. 占空比更新时 PWM 是否稳定

预期波形：

```text
25% 120ms -> 0% 180ms
50% 120ms -> 0% 180ms
75% 120ms -> 0% 180ms
100% 120ms -> 0% 180ms
0% 保持 1500ms
下一轮
```

实际观察重点：

- 示波器占空比是否按四档变化
- 手感上是否能感受到由弱到强
- 100% 档位是否明显最强

适合场景：

- 做后续“不同反馈强度档位”映射前的基础验证

---

## 5.5 `RUMBLE_TEST_CASE_SAFETY_CLAMP`

作用：

- 验证强度钳位和持续时间钳位是否生效

测试会做什么：

- 故意请求超过 `100%` 的强度
- 故意请求超过上限的持续时间
- 但代码实际输出时会自动裁剪

它验证的功能点：

1. 错误强度参数是否会被钳位到 `100%`
2. 错误持续时间是否会被钳位到 `300ms`
3. 保护后系统是否还能正常执行测试

预期日志重点：

```text
safety clamp active: requested level XXX%, forced to 100%
safety clamp active: requested active XXX ms, forced to 300 ms
```

预期波形：

```text
不是超范围占空比
而是 100% 占空比

不是超长持续时间
而是 300ms 持续时间
```

适合场景：

- 验证保护逻辑
- 防止测试参数写错导致马达长时间满功率运行

## 7. 如何启用振动马达测试入口

当前入口文件：

- [demo_sle_uart.c](/data/tr531x_proj/projects/light_gun_driver_test/sdk_overlay/custom/demo_sle_uart.c)

默认配置是：

- 电磁铁测试启用
- 振动测试关闭

如果你要只测振动马达，建议改成：

```c
#define LIGHT_GUN_ENABLE_SOLENOID_TEST 0
#define LIGHT_GUN_ENABLE_RUMBLE_TEST 1
```

不建议默认同时开两套测试，原因很简单：

- 两路一起动作会影响听感判断
- 电源波动也可能互相干扰

## 8. 关键参数怎么调

文件：

- [rumble_test.h](/data/tr531x_proj/projects/light_gun_driver_test/sdk_overlay/custom/rumble_test.h)

常用参数：

- `RUMBLE_TEST_PWM_CHANNEL`
  切换 PWM 通道

- `RUMBLE_TEST_PWM_GROUP_ID`
  切换 PWM 组
  这个参数在 `CONFIG_PWM_USING_V151=y` 时尤其重要，当前实现会先把 channel 挂到 group 上，再启动 group。

- `RUMBLE_TEST_PWM_PIN`
  切换 PWM 引脚

- `RUMBLE_TEST_PWM_PIN_MODE`
  切换引脚复用模式

- `RUMBLE_TEST_LEVEL_PERCENT`
  默认振动强度

- `RUMBLE_TEST_ACTIVE_MS`
  单次振动持续时间

- `RUMBLE_TEST_GAP_MS`
  两次振动之间的停顿

- `RUMBLE_TEST_PERIOD_MS`
  周期振动模式的一轮周期

- `RUMBLE_TEST_AUTOFIRE_PULSES`
  一轮连续振动次数

- `RUMBLE_TEST_MAX_ACTIVE_MS`
  单次振动最大时间钳位

## 9. 推荐测试顺序

建议按这个顺序：

1. `SINGLE_PULSE_ONCE`
   先确认 PWM 输出、接线、单次振动是否正确。

2. `SAFETY_CLAMP`
   先确认保护逻辑有效。

3. `PERIODIC_PULSE`
   检查周期稳定性。

4. `DUTY_SWEEP`
   检查强度映射。

5. `BOUNDED_AUTOFIRE`
   检查多脉冲节奏和休息机制。

## 10. 异常现象与排查建议

## 10.1 上电后马达一直转

可能原因：

- 驱动板默认上电状态就拉高
- PWM 引脚模式配置错误
- 驱动极性理解反了

先查：

- PWM 输出脚是否上电即存在高占空比
- 驱动板输入脚极性

## 10.2 日志显示有振动，但马达不转

可能原因：

- PWM 只到主控输出，没有真正到驱动板
- 驱动板没供电
- 电机供电和主控地没有共地
- PWM 通道和引脚映射不对
- PWM group 没配置对，导致 V151 路径没有真正启动

先查：

- PWM 输出脚是否有占空比变化
- 驱动板输入脚是否同步变化
- 马达供电是否正常

另外重点看日志里有没有这些错误：

- `uapi_pin_set_mode failed`
- `uapi_pwm_open failed`
- `uapi_pwm_set_group failed`
- `uapi_pwm_start_group failed`
- `uapi_pwm_update_cfg failed`

如果你现在是“完全没波形”，优先怀疑：

1. PWM group 没起起来
2. pin mode 选错
3. 通道和引脚不是同一路 PWM

## 10.3 振动有，但强弱几乎没差别

可能原因：

- 驱动板对 PWM 响应不明显
- 马达惯性太大，短时变化不明显
- 占空比变化范围太小

建议：

- 优先测 `DUTY_SWEEP`
- 把不同档位差距拉大
- 用示波器先确认占空比真的变了

## 10.4 振动时间明显比配置长

可能原因：

- 软件定时器调度存在延迟
- 观察的是马达机械惯性，而不是 PWM 本身
- 驱动板对停止时有余振

先查：

- PWM 输出脚实际波形
- 马达停转晚于 PWM 停止，通常是机械惯性，不一定是代码错

## 10.5 周期性测试里偶发多震或漏震

可能原因：

- 测试入口同时启动了别的反馈任务
- 定时器状态机被其他任务影响
- 电源不稳定

先查：

- `demo_sle_uart.c` 是否只启用了 rumble 测试
- 日志里的 `pulse x/y`

## 10.6 马达太热

可能原因：

- 单次持续时间太长
- 周期太密
- 长期满占空比运行

建议：

- 降低 `RUMBLE_TEST_LEVEL_PERCENT`
- 减少 `RUMBLE_TEST_ACTIVE_MS`
- 增大 `RUMBLE_TEST_GAP_MS`
- 增大 `RUMBLE_TEST_CYCLE_REST_MS`

## 11. 最后建议

振动马达和电磁铁最大的区别在于：

- 电磁铁更像“打一下”的数字脉冲
- 振动马达更像“持续一段时间的强度输出”

所以你后续二次开发时，建议把振动控制拆成三层：

1. PWM 输出层
   只负责占空比写入

2. 时序层
   只负责“振动多久、停多久”

3. 业务层
   决定单次振动、连续振动、强度 sweep 还是其他效果

你现在这份测试代码，本质上已经把这三层的最小闭环验证搭好了，后续可以直接拿来当驱动验证基线。
