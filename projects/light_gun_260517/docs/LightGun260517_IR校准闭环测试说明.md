# LightGun260517 IR 校准闭环测试说明

本文档说明当前 `projects/light_gun_260517` 里 IR 分阶段校准协议怎么发命令、怎么看返回，方便你马上联调上位机。

## 1. 适用范围

当前校准协议位于：

- [proto_docked.c](/data/tr531x_proj/projects/light_gun_260517/sdk_overlay/custom/src/protocols/proto_docked.c)
- [svc_calibration.c](/data/tr531x_proj/projects/light_gun_260517/sdk_overlay/custom/src/services/svc_calibration.c)

这版已经从“只做简单中心校准”升级成“分阶段校准协议”。

推荐上位机以后都按分阶段方式做，不要再只依赖旧的中心点思路。

## 2. 校准阶段定义

协议里的阶段值和 `of_cal_state_t` 一致：

1. `0x00` = `IDLE`
2. `0x01` = `TOP`
3. `0x02` = `BOTTOM`
4. `0x03` = `LEFT`
5. `0x04` = `RIGHT`
6. `0x05` = `CENTER`
7. `0x06` = `VERIFY`
8. `0x07` = `DONE`

其中真正采样阶段是：

- `TOP`
- `BOTTOM`
- `LEFT`
- `RIGHT`
- `CENTER`

`VERIFY` 表示五个阶段样本已经采完，结果处于“待确认、待提交”状态。  
`DONE` 表示已经正式提交保存。

## 3. 返回包统一格式

校准相关返回包统一使用 12 字节：

```text
byte0  : 响应命令
byte1  : result，0=成功，1=失败
byte2  : 当前阶段 state
byte3  : center_x low
byte4  : center_x high
byte5  : center_y low
byte6  : center_y high
byte7  : top
byte8  : bottom
byte9  : left
byte10 : right
byte11 : 保留，当前固定 0
```

注意：

- `center_x / center_y` 是 16 位小端
- `top / bottom / left / right` 当前按 8 位返回
- 如果处于 `VERIFY`，这里返回的是“待提交参数”
- 如果已经 `DONE`，这里返回的是“已保存参数”

## 4. 命令定义

### 4.1 `0x50` 开始校准

作用：

- 清空本轮待校准数据
- 状态切到 `TOP`

发包：

```text
50
```

返回：

- `0xD0`

用途：

- 这是整套校准流程的入口

### 4.2 `0x51` 提交采样点

支持两种格式。

格式 A，兼容旧方式，不显式指定阶段：

```text
51 x_lo x_hi y_lo y_hi
```

含义：

- 把当前 `(x, y)` 样本送给“当前阶段”
- 当前阶段由设备内部状态机决定

格式 B，推荐的新方式，显式指定阶段：

```text
51 stage x_lo x_hi y_lo y_hi
```

含义：

- 先把当前阶段切到 `stage`
- 再把 `(x, y)` 样本送入该阶段

返回：

- `0xD1`

注意：

- 每个阶段内部需要累计 `16` 个样本
- 采满后会自动切到下一阶段
- `CENTER` 采满后会自动进入 `VERIFY`

### 4.3 `0x52` 提交校准结果

作用：

- 只有当前状态是 `VERIFY` 时才能成功
- 成功后把中心点和四边参数正式写入 profile
- 状态切到 `DONE`

发包：

```text
52
```

返回：

- `0xD2`

### 4.4 `0x53` 查询当前校准状态

发包：

```text
53
```

返回：

- `0xD3`

用途：

- 上位机可随时轮询当前阶段和当前暂存结果

### 4.5 `0x54` 强制切换当前阶段

发包：

```text
54 stage
```

返回：

- `0xD4`

用途：

- 用于上位机明确控制“现在开始采 top / bottom / left / right / center”
- 也适合调试时手动纠偏

注意：

- 允许切到 `TOP~CENTER`
- 也允许切到 `VERIFY`
- 不允许直接切到 `IDLE` 或 `DONE`

### 4.6 `0x55` 退出/放弃本轮校准

发包：

```text
55
```

返回：

- `0xD5`

作用：

- 清空本轮待校准数据
- 状态回到 `IDLE`

## 5. 推荐的闭环测试流程

这是最适合上位机联调的流程。

### 5.1 开始校准

发送：

```text
50
```

期望：

- 返回 `D0`
- `result=0`
- `state=TOP`

### 5.2 采 `TOP`

做法：

- 屏幕上提示用户瞄准上边中点
- 连续取 16 个原始 IR 点
- 每个点用显式阶段方式发送

示例：

```text
51 01 34 12 78 56
```

表示：

- 阶段=`TOP`
- x=`0x1234`
- y=`0x5678`

期望：

- 前 15 个样本返回时，大概率 `state` 还是 `TOP`
- 第 16 个样本返回后，状态应自动切到 `BOTTOM`

### 5.3 采 `BOTTOM`

同上，把阶段值改成 `0x02`。

第 16 个样本完成后，状态应切到 `LEFT`。

### 5.4 采 `LEFT`

同上，把阶段值改成 `0x03`。

第 16 个样本完成后，状态应切到 `RIGHT`。

### 5.5 采 `RIGHT`

同上，把阶段值改成 `0x04`。

第 16 个样本完成后，状态应切到 `CENTER`。

### 5.6 采 `CENTER`

同上，把阶段值改成 `0x05`。

第 16 个样本完成后，状态应切到 `VERIFY`。

### 5.7 查询暂存结果

发送：

```text
53
```

期望：

- 返回 `D3`
- `state=VERIFY`
- `center_x / center_y / top / bottom / left / right` 都已经有值

这一步的意义：

- 上位机可以先显示结果
- 如果结果明显异常，可以不提交，重新来一轮

### 5.8 正式提交

发送：

```text
52
```

期望：

- 返回 `D2`
- `result=0`
- `state=DONE`

表示这套校准参数已经正式写入 profile。

## 6. 上位机联调建议

建议 PC 端把流程做成明确的五步向导：

1. 点击“开始校准”，发 `0x50`
2. 提示用户瞄准 `TOP`，采 16 点
3. 再依次提示 `BOTTOM / LEFT / RIGHT / CENTER`
4. 发 `0x53` 展示暂存结果
5. 用户确认后发 `0x52`

这样做的好处：

- 业务流程和固件状态机是一一对应的
- 调试时一眼就能看出卡在哪个阶段
- 后面做失败重试也简单

## 7. 失败场景怎么看

### 7.1 `0x51` 返回失败

常见原因：

- 当前不在采样阶段
- 你还没先发 `0x50`
- 阶段值发错

建议：

- 先发 `0x53` 看当前 state
- 必要时发 `0x55` 放弃，再重新 `0x50`

### 7.2 `0x52` 返回失败

常见原因：

- 还没进入 `VERIFY`
- 五个阶段没采完

判断方法：

- 发 `0x53`
- 看 `state` 是否已经到 `0x06`

### 7.3 某个阶段采样明显漂

建议：

- 上位机在每阶段采样前先清楚提示目标点位
- 一旦样本点跳得太厉害，不要立刻提交
- 可以直接 `0x55` 放弃重来

## 8. 和旧协议的兼容关系

这版保留了旧的 `0x50~0x53` 命令族，没有破坏老入口。

但建议你后续上位机统一升级到“显式阶段模式”：

- 用 `0x54` 或 `0x51 + stage`
- 不要依赖设备“猜你现在采的是哪一阶段”

这样闭环联调最稳，也最方便后面排查问题。
